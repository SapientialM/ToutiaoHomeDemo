## 4. 数据流设计

### 4.1 网络数据流

```
App 启动 / Tab 切换 / 下拉刷新
    ↓
ViewModel 发送 Refresh 请求
    ↓
Repository 调用 RemoteDataSource
    ↓
Retrofit → OkHttp → Server API
    ↓
返回 DTO → Mapper 转换为 Domain Model
    ↓
写入 Room 数据库（Paging RemoteMediator）
    ↓
Room Flow 触发 UI 更新
```

### 4.2 本地数据流（离线场景）

```
无网络 / 数据库有缓存
    ↓
Repository 直接读取 Room
    ↓
Flow 发射缓存数据
    ↓
UI 展示缓存内容 + 提示"已展示缓存数据"
```

## 7. 卡片组件设计

### 7.1 异构列表策略

使用密封类 + when 表达式实现类型安全的多卡片渲染：

```kotlin
sealed class FeedCard {
    abstract val id: String

    @Stable
    data class TextTop(...): FeedCard()
    @Stable
    data class LeftTextRightImage(...): FeedCard()
    @Stable
    data class LargeImage(...): FeedCard()
    @Stable
    data class Video(...): FeedCard()
}
```

### 7.2 LazyColumn 渲染

```kotlin
LazyColumn {
    items(
        items = feedItems,
        key = { it.id }
    ) { card ->
        when (card) {
            is FeedCard.TextTop -> TextTopCard(card)
            is FeedCard.LeftTextRightImage -> LeftTextRightImageCard(card)
            is FeedCard.LargeImage -> LargeImageCard(card)
            is FeedCard.Video -> VideoCard(card)
        }
    }

    // 加载更多 Footer
    if (isLoadingMore) {
        item { LoadingFooter() }
    }
}
```

## 8. 分页策略（Paging3 + RemoteMediator）

### 8.1 分页流程

```
用户滑动列表
    ↓
LazyColumn 接近底部
    ↓
PagingSource 触发 LoadType.APPEND
    ↓
RemoteMediator 检查网络
    ↓
网络请求下一页数据
    ↓
写入 Room 数据库
    ↓
Room Flow 自动刷新 UI
```

### 8.2 刷新策略

```
用户下拉
    ↓
PullRefresh 触发
    ↓
RemoteMediator LoadType.REFRESH
    ↓
清空本地数据
    ↓
请求第一页数据
    ↓
写入 Room
    ↓
UI 回到顶部并展示新数据
```

## 11. 安全与异常处理

### 11.1 异常处理策略

```
网络请求异常
    → 返回 Error State，展示重试按钮
    → 如有本地缓存，展示缓存 + 弱网提示

JSON 解析异常
    → 记录日志，返回 Error State
    → 使用 Kotlinx Serialization 严格模式提前发现

数据库异常
    → 降级为仅网络模式
    → 提示用户"缓存异常，已切换在线模式"
```

### 11.2 日志策略

- 使用 Timber 进行日志管理
- Debug 模式：详细日志
- Release 模式：仅错误日志

## 12. 答辩高频问题预案

### Q1：为什么选择 MVI 而不是 MVVM？
**A**：信息流状态复杂（Loading/Refreshing/LoadingMore/Error/Empty），MVI 的密封类 UiState 能精确描述所有状态，单向数据流更利于调试和状态追溯。在字节内部，复杂页面也倾向使用类似 MVI 的严格状态管理。

### Q2：Paging3 的 RemoteMediator 工作原理？
**A**：RemoteMediator 是 Paging3 提供的网络+本地混合分页方案。它在 PagingSource 加载数据时介入，负责网络请求并将结果写入 Room。PagingSource 实际只从 Room 读取，Room 的 Flow 特性会自动通知 UI 更新。

### Q3：如果后续需求要求支持视频自动播放，如何扩展？
**A**：
1. 引入 ExoPlayer 作为视频播放引擎
2. 在 `VideoCard` 组件中嵌入 PlayerView
3. 通过 `LazyListState.layoutInfo.visibleItemsInfo` 检测可见视频卡片
4. 在 ViewModel 层管理 Player 实例池（避免每个卡片创建独立 Player）
5. 滑动时自动暂停不可见视频，播放当前可见视频

### Q4：Compose 相比传统 RecyclerView 的优势和劣势？
**A**：
- **优势**：声明式写法更简洁，状态驱动 UI 减少样板代码，与 Kotlin 协程/Flow 集成更自然
- **劣势**：重组机制需要手动优化（使用 key、remember 等），生态成熟度略低于 RecyclerView
- **趋势**：Google 和字节内部都在全面迁移 Compose，是长期技术方向
